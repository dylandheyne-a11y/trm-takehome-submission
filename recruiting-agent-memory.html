Recruiting Agent Memory Summary

Privacy note: this is a sanitized summary of the agent memory used to guide the TRM review. It removes prior candidate names, hiring manager names, company-specific private examples, and historical search details.

What The Memory Does

The memory is the judgment layer behind the profile evaluator. The rubric gives structure, but the memory teaches the agent how recruiters actually calibrate: what to trust, what to question, and when a profile is adjacent to the work rather than doing the work.

Core Principles

  • Verdict first. Start with Strong Yes / Yes / Soft Flag / No / Def No, then explain why.
  • Score is not truth. Scores help rank a pool, but a high score from a weak or wrong-archetype pool is still not enough.
  • Company is a prior, not a verdict. Target companies help decide who enters review. Candidate evidence decides whether they advance.
  • Function beats industry. Similar industry does not matter if the person did the wrong work.
  • Thin profiles require probes. A sparse LinkedIn from a strong company should become a validation item, not an invented positive or an automatic reject.
  • HM feedback updates the model. Every clear yes/no reason becomes a rule candidate for the next batch.

False Positive Pattern

Most searches develop repeat false positives within the first 10-15 profiles. The agent is instructed to identify and log those patterns quickly.

Surface signalRiskWhat the agent should ask
"Worked with APIs"May be API consumer, not API builder.Did they design/own the API surface?
"Platform engineer"Could mean infra, tooling, or product platform.Was the work customer/developer-facing?
"Full-stack"May be frontend-heavy with limited backend ownership.What backend services, data models, or APIs did they own?
"Security / compliance"Could be analyst, GRC, DevOps, or product engineering.Were they building product systems or supporting controls?
"High-growth company"Company quality can mask wrong function.Does the profile evidence match the role, independent of company brand?

TRM-Specific Application

  • Prioritize product/API/backend ownership over generic engineering seniority.
  • Separate fraud/risk/security product engineers from ML infra or data-only profiles.
  • Look for public API, REST/GraphQL, SDK, auth, identity, risk, fraud, compliance workflow, and platform product evidence.
  • Treat startup pace as operating-context evidence, not a substitute for technical fit.
  • Use outreach probes to validate missing LinkedIn details instead of over-scoring thin profiles.

How This Was Used

For the TRM search, this memory shaped the calibration loop: Codex scored candidates, Claude/human review challenged misses, and the evaluator rules were tightened around function, evidence quality, and false-positive caps.

The public calibration artifact shows that process without exposing the full private memory.
