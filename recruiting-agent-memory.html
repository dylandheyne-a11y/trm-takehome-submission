Recruiting Agent — Memory & Methodology

Built from live resume review sessions across 6+ roles, 100+ candidate evaluations

Author: Dylan Heyne | For use in Claude Code / Codex agent build

---

WHAT THIS IS

This document captures what has actually worked across multiple live recruiting projects. It is NOT a rubric document. It is a reasoning document.

The rubric is a scaffold. This is the judgment layer that runs on top of it. Feed both to the agent. The rubric structures output. This document prevents the agent from being wrong in systematic ways.

---

THE CORE PRINCIPLE

The scoring system is a forcing function, not a truth machine.

Scores rank candidates against each other. They do not tell you whether any candidate is actually hireable. The agent's job is to surface the top of the pool AND flag when the pool is structurally weak.

A score of 95 from a weak pool is still a weak candidate. Always calibrate against absolute bar, not relative rank.

---

WHAT HAS WORKED

1\. Verdict First, Always

Every candidate response leads with a clear decision before any explanation.

  • Strong Yes / Yes / Soft Flag / No / Def No
  • One line justification follows
  • Evidence bullets after that
  • Never bury the decision in qualifications

This prevents hedge outputs that look useful but give no direction.

2\. The False Positive Pattern Log

Every role develops 2–3 false positive archetypes within the first 10–15 profiles. These are profiles that look right at a glance but fail on the critical filter.

Proven false positive patterns by role type:

RoleSurface SignalFalse Positive Pattern
SDK Engineer"Android developer"App developer using SDKs, not building them
Quant Risk"Execution quant"TCA analyst, desk strategist, model validator — none build execution infrastructure
FP\&A L6"Full P\&L ownership"Revenue/GTM finance; OpEx was "supported," not owned
Recon Ops"Reconciliation experience"AR/AP aging, budget variance — not transaction/position recon with break lifecycle
Cyprus Accountant"CIF experience"Audited CIF clients vs. worked inside one — different domain knowledge
IT Audit"IT audit experience"Financial audit with IT controls exposure ≠ IT/security audit practitioner

Agent instruction: Build a false positive log per role from the first 15 profiles reviewed. Update it when HM feedback reveals a new pattern. Surface the log when evaluating ambiguous profiles.

3\. The Critical Language Test

Claims must be specific and use practitioner language. Generic language \= surface-level exposure.

Examples of the distinction:

Weak (surface exposure)Strong (practitioner)
"Managed reconciliations""Aged breaks, root cause analysis, escalation workflow, daily safeguarding report"
"Full-stack engineer""Built public API product with Postgres-backed services and Node.js"
"OpEx experience""Owned $45M OpEx budget for Engineering, built zero-based model from scratch"
"Risk quant""Built stochastic liquidation model adapting Almgren-Chriss for stressed regimes"
"Internal audit""Led end-to-end ITGC audits across cloud infrastructure; CISA certified"

If a profile uses generic language, treat it as unexplained, not evidenced. The agent should flag specific language gaps, not assume positive intent.

4\. Calibration Loop — How to Update Mid-Search

The most effective sessions used real-time calibration. After the first 10–15 reviews:

  • Identify the 2–3 strongest candidates
  • Extract the keywords and company patterns that made them strong
  • Tighten boolean strings to match those patterns
  • Log the discard patterns so they inform future reviews

Calibration trigger: When 3+ candidates in a row are discarded for the same reason, that reason should become an explicit exclusion in the search criteria, not just a review filter.

5\. The Tenure Pattern Rule

Short tenures are only concerning when they are a pattern, not a data point.

  • Single short tenure in early career: acceptable
  • Multiple tenures under 18 months across career: flag
  • Single long tenure as only employer: flag opacity risk (not enough calibration data)

For senior roles: 2+ years per company as the expectation. One exception allowed for early career.

6\. Company as Prior, Not Verdict

Target company \= screening shortcut, not qualification. The most consistent failure mode is over-weighting company name.

The rule from RevenueCat / Vanta profiles: Vanta appeared twice in TRM review. Neither profile surfaced backend/API evidence. Company got them into review. Evidence (or lack of) determined the outcome.

Do not advance candidates solely because they work at a target company. Do not reject candidates solely because they don't.

7\. The Opacity Problem

LinkedIn profiles at high-quality startups are often thin. Engineers at Vanta, Linear, Ramp, Stripe frequently write minimal bios. A thin profile from a strong company ≠ weak candidate.

Agent instruction: When a profile is opaque from a Tier 1 company, flag for recruiter validation rather than auto-disqualifying. Advance to outreach with a probe question in the message. Do not score penalize blindly for missing keywords that strong engineers typically don't write about.

The probe approach: Outreach message designed to surface technical depth. If reply confirms backend/API/product ownership: advance. If reply is surface-level: pass.

8\. HM Feedback Integration — The Most Valuable Signal

Every HM "no" with a reason is worth more than 20 resume reviews.

From Quant Risk (Xiadi Liu):

  • "They calculate when to liquidate, not how to execute the liquidation" → updated boolean to exclude TCA analysts
  • "Builders, not consultants" → added execution infrastructure keywords, penalized advisory titles

From FP\&A (Taylor Hankes):

  • Varda Malhotra pushed back: claimed full P\&L but it was revenue-side → rule: require dollar amounts AND explicit OpEx language, not just P\&L

From Recon Ops (Kaiyi Gan):

  • Hilton Otamere: 16-month tenure, borderline experience → HM advanced → rule: language quality and domain specificity can outweigh tenure length for ops roles

Agent instruction: Log every HM decision with reason. After 5 HM decisions, surface calibration delta: "HM has rejected 3 candidates for \[reason\]. Recommend updating search criteria to exclude \[pattern\]."

WHAT HAS NOT WORKED

1\. Starting with Tight Booleans

Every role where search started tight returned too-small pools and missed qualified candidates hiding behind different keywords. Always start broad, sample 10–15 profiles, then tighten.

2\. Over-Relying on Score

Scores are for ranking. They don't replace judgment on:

  • Whether the pool is strong enough to hire from
  • Whether a borderline profile has hidden depth
  • Whether a strong score is on a structurally weak profile

A 95 in a weak pool is still a pass.

3\. Industry Filter as Proxy for Role Fit

"Fintech experience" does not mean "recon experience." "Crypto experience" does not mean "execution quant experience." "IT background" does not mean "IT audit experience." Industry is context. Function is the filter. Always confirm function before industry.

4\. Treating Penalizing as Disqualifying

Penalties in the rubric are risk flags, not automatic DQs. Opacity \+ strong company \= advance with probe. Short early tenure \+ strong recent pattern \= advance. Missing certification \+ strong operational evidence \= discuss with HM, don't auto-DQ.

5\. Skipping the "What They Actually Built" Question

The single most common error: advancing a profile where the candidate adjacent to the target work rather than doing the target work.

  • Used the SDK vs. built the SDK
  • Audited the recon team vs. did the reconciliation
  • Analyzed the liquidation risk vs. built the liquidation algorithm
  • Supported the OpEx model vs. owned the OpEx model

Always ask: did they do the thing, or were they near the thing?

STACK RANKING RULES

  1. Rank within role, not across roles. Context is everything.
  2. When two candidates are within 5 points, the tiebreaker is hireability and tenure pattern, not score.
  3. Always note whether the stack rank is from a strong or weak pool. A \#1 in a weak pool needs that label.
  4. Update the rank every 10 candidates reviewed. Don't lock it after 3\.
  5. If the top candidate has a major unknown, surface it before presenting to HM.

ROLE-SPECIFIC CONTEXT: TRM LABS FULL-STACK ENGINEER

The Critical Distinction for This Role

API builder vs. API consumer. Same false positive risk as SDK authorship vs. SDK integration.

An engineer who consumed APIs, integrated third-party services, or called external endpoints ≠ an engineer who designed and owned a public API surface.

Practitioner language to look for:

  • "Designed REST API," "built API product," "public API," "developer platform," "API gateway"
  • "PostgreSQL schema design," "query optimization," "database performance"
  • "Technical anchor," "tech lead," "owned feature end-to-end from requirements to launch"

False positive archetypes for this role:

  • Frontend-heavy engineers with backend exposure but no API ownership
  • Data engineers who touch Postgres but don't build product APIs
  • Infrastructure engineers who maintain API infrastructure but didn't design it
  • Early career engineers at strong companies with insufficient scope

Hard Filters (Non-Negotiable)

  • 5+ years professional SWE experience
  • US-based and eligible
  • At least one high-growth startup stint
  • Not from direct TRM competitors (Chainalysis, Elliptic, Merkle)

AI Fluency Bar

TRM explicitly evaluates this in process. Surface it in outreach. Candidates who haven't integrated AI into their workflow will likely DQ in process — save everyone's time.

Mission Fit as a Screen

TRM's JD explicitly warns the pace/intensity isn't for everyone. This is a culture screen, not a perk section. Use it in outreach to self-select. The mission (crypto intelligence / law enforcement) resonates strongly with specific candidate profiles:

  • Govtech / national security adjacent engineers
  • Engineers at fraud/compliance/risk platforms (Vanta, Drata, Persona, Socure, Sardine)
  • Engineers who explicitly write about impact, not just craft

Company Signal Priority

Tier 1: Chainalysis, Elliptic, Vanta, Ramp, Brex, Mercury, Modern Treasury, Sardine, Stripe, Plaid Tier 2: Fireblocks, Alchemy, Anchorage, QuickNode, ComplyAdvantage, Persona, Alloy Tier 3: Any API-first B2B SaaS with Node.js \+ Postgres signal

AGENT OPERATING INSTRUCTIONS

On receiving a profile:

  1. Check hard filters first. If any fail, DQ immediately with reason.
  2. Apply role-specific false positive check before scoring.
  3. Score using rubric.
  4. Apply penalties.
  5. Issue verdict.
  6. Note top 1–2 unknowns.
  7. Suggest outreach hook or next step.
  8. Update stack rank.

On HM feedback:

  1. Log decision \+ reason.
  2. Check if it matches an existing discard pattern.
  3. If new pattern: add to false positive log, flag for boolean update.
  4. If confirms existing pattern: reinforce discard signal in future evals.

On pool quality:

  1. After 15 profiles, surface pool quality assessment.
  2. If top candidate has meaningful gaps: say so.
  3. If pool is structurally weak: recommend boolean/search adjustment before advancing weak candidates.

On ambiguous profiles (Opacity Rule):

  1. Flag as Skip if insufficient data from a Tier 1 company.
  2. Recommend outreach with probe question.
  3. Do not score penalize for missing keywords that strong engineers typically omit.
  4. Revisit after outreach response.

---

CALIBRATION CHECKPOINTS

Run these at defined intervals, surface to recruiter:

After 15 profiles:

  • Top 3 candidates so far
  • False positive pattern identified (if any)
  • Boolean refinement recommendation (if any)
  • Pool strength assessment

After first 3 HM decisions:

  • HM calibration summary: what they're saying yes/no to and why
  • Delta from original criteria: what's shifted
  • Recommended screening question updates

After 5 HM decisions:

  • Pattern confirmation or contradiction
  • Recommend search strategy adjustment if \>60% pass rate or \<20% advance rate

---

META-LESSON

The review process works because it holds pattern recognition across many evaluations and calibrates in real time.

A static rubric run once per candidate is a tool. A calibrating agent that learns from the session is intelligence.

The rubric structures the output. The false positive log prevents systematic errors. The HM feedback loop closes the gap between what the job description says and what the hiring manager actually wants. The stack rank forces relative judgment, not just absolute assessment.

All four together \= the system. None alone is sufficient.
