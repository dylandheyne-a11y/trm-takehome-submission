What is missing

  • Explicit backend/API product ownership.
  • Node.js / TypeScript / Postgres.
  • Clear startup-velocity evidence beyond Vanta company signal.

False-positive pattern

API-adjacent / client-feature builder vs. API product builder.

This mirrors the SDK search pattern:

> adjacent to the work is not the same as doing the work.

Vanta Pattern After 3 Profiles

Vanta remains a legitimate source for:

  • compliance/security adjacency,
  • enterprise trust,
  • fast-growth operating environment,
  • likely mission resonance.

But Vanta alone does not advance a candidate.

Observed pattern:

  • Vanta profiles are often thin on stack and ownership.
  • Company signal is strong, but candidate-level evidence varies.
  • Valyo is stronger because tech lead/founding engineer/activity-feed evidence gave additional signal.
  • Isabelle and Andrew need validation on stack depth and seniority/ownership before advancing.

Working rule:

> Vanta is a strong company prior, not a verdict. Advance only when Vanta is paired with team/function signal, senior scope, and evidence or probe-confirmed backend/API/product ownership.

Calibration Event 3: Full Codex Tool Review

Source: TRM_CODEX_CALIBRATION_2026-05-21.md and trm-candidate-review-log-2026-05-21.csv

System-Level Fixes

The tool was still making four systematic errors:

  1. Keyword scoring over functional judgment.
  2. Company tier treated as evidence rather than review prior.
  3. False-positive flags did not cap verdicts.
  4. Frontend engineers and AI infra/FDE profiles could still score too high.

Working rule:

> Functional archetype comes before score. If the archetype is wrong, keywords cannot rescue the profile.

Rubric Updates

  • Classify functional archetype before score:

- product/API/full-stack

- fraud/risk/security product

- ML/data/infra

- frontend

- non-engineering/GTM

- unclear

  • Active false-positive flag caps verdict at Soft Flag unless the profile explicitly resolves it.
  • Frontend-primary identity gets a heavy penalty and cannot advance for this role.
  • Fraud/risk/security product engineering is a valid TRM archetype and should not trigger ML/FDE caution.
  • Generic phrases like "backend experience" or "worked with APIs" are weak without ownership language.
  • Practitioner phrases like "owned public API," "designed REST API surface," "full product lifecycle," "fraud detection," "risk scoring," and "SDK ownership" should carry more weight.
