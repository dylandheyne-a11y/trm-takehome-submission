What The CSV Showed

The workstation was useful for logging, but too generous on keyword scoring.

Main issues:

  1. Default company source was too favorable.

- The tool defaulted to Primary company target, so unknown or mis-parsed companies received a startup/company prior they did not earn.

  1. Parser grabbed noisy LinkedIn text as company/title.

- Example: Christopher Wirick logged as Distinguished Software Engineer at HubSpot at Unity, which did not match the actual review context.

- Example: David Oy company parsed as a sentence fragment from profile text.

  1. Keyword scoring over-rewarded general words.

- API, platform, product, service, built, designed appeared in pasted profile text, but the tool did not distinguish whether the candidate owned product APIs or was adjacent to them.

  1. AI infra / Forward Deployed profiles were misclassified.

- Christopher Wirick, David Oy, Ian Carrasco, and Alex Ker were over-scored locally relative to Claude calibration.

- Baseten/FDE profiles looked technical but skewed ML infra, inference, performance, deployment, and GTM-adjacent work.

Calibration Update Applied

Tool changes:

  • Default company source is now Unknown / needs research.
  • Baseten is now Deprioritized for this search.
  • ML infra/FDE/inference patterns now trigger a caution flag.
  • ML infra/FDE profiles cannot become clean Yes unless product/API ownership is explicit.
  • Baseten/FDE false-positive pattern added to calibration log.

May 21 Calibration Update Applied

Source: /Users/dylanheyne/Downloads/TRM_CODEX_CALIBRATION_2026-05-21.md

Additional tool changes:

  • Company/startup signal was reduced to a 10-point prior. It can get a profile into review, but should not carry the verdict.
  • Frontend-primary identity now gets a heavy penalty and a Def No cap for this TRM API/backend search.
  • GTM/content/investing patterns now get a Def No cap unless sustained SWE work is visible.
  • API consumer language now gets penalized and capped unless there is explicit API builder evidence.
  • Fraud/risk/security product engineering is now treated as a positive TRM archetype, not ML infra.
  • Practitioner evidence such as REST API surface design, SDK ownership, gRPC gateway, credits/billing services, Postgres datastore, fraud/risk work, full product lifecycle, reconciliation engines, and low-latency market connectors now receives a boost.
  • Logged CSV rows now include richer analysis fields: archetype, built-analysis, false-positive risk, unknowns, hireability read, next step, probe, and full summary.

Working rule:

> Company is a sourcing prior. Functional archetype is the gate. Evidence is the verdict.

Working Rule Going Forward

Do not let keyword hits outrun role archetype.

For TRM:

> API/backend/platform keywords matter only when they support product ownership, not when they appear in ML infra, customer deployment, or implementation-adjacent contexts.
